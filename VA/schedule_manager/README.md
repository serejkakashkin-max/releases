# Schedule Manager

Изолированный VA-модуль для работы с графиками дежурств.

- Точка входа в основном приложении: `/admin/va/schedule-manager`

## Как подключается

Модуль регистрируется опционально через основной Flask app. Флаг:

```json
{
  "modules": {
    "va_schedule_manager": {
      "enabled": true
    }
  }
}
```

Если папка `VA/schedule_manager` отсутствует или флаг выключен, основной проект продолжает запускаться.

## Административный доступ

Доступ закрыт backend-проверкой admin-session. Для входа нужен `sup_admin_token`, а для безопасной session должен быть задан один из параметров:

- environment: `SUP_ADMIN_SESSION_SECRET`
- `config.json`: `sup_admin_session_secret`

Token не хранится в session. Для изменяющих запросов используется CSRF заголовок `X-CSRF-Token`.

## Runtime-данные

Рабочие данные не хранятся внутри `VA/`.

```text
cache/va_schedule_manager/
├── data/
├── uploads/
├── exports/
└── state/
```

Locks, backups и migration reports находятся в `cache/va_schedule_manager/state/`.

## Центральный справочник сотрудников

Источник сотрудников выбирается consumer mode `va_schedule_manager` в СУП-параметрах:

- `legacy` — VA полностью использует собственный `employees.json`;
- `compare` — списки и порядок сравниваются, рабочее поведение остаётся legacy;
- `directory` — общие контакты читаются из `employee_directory.json`.

В режиме `directory` существующее VA-имя сохраняется как ключ строк графика. Email, телефон, локация и табельный номер подставляются из центрального справочника только в память и не копируются обратно в VA runtime. Статус, роль, компетенции и готовность к сверхурочной работе по-прежнему хранятся и редактируются в VA. Добавление, удаление и изменение общих полей выполняется через центральный редактор СУП.

Новый central-сотрудник с membership VA появляется в новых графиках сразу. До создания локальной VA-настройки он получает безопасные defaults: активен, компетенция `support`, обычная роль и готовность к сверхурочной работе. При первом изменении VA-настроек создаётся локальная shadow-запись; центральные контакты остаются источником истины.

При создании нового месяца по умолчанию используется текущий справочник активных сотрудников. Вариант «Из последнего графика» сохраняется для случаев, когда нужно намеренно повторить прежний состав без новых сотрудников.

Форма добавления в VA остаётся доступной для обратной совместимости. В режиме `directory` она создаёт центрального сотрудника только с membership `va_schedule_manager` либо включает этот membership у уже существующей identity. Участие в остальных сервисах остаётся выключенным и настраивается отдельно в СУП. Конкурентное изменение справочника определяется по revision и ETag; при конфликте запись не перезаписывается.

Изменение центрального порядка и чистое добавление/исключение VA-membership являются штатными операциями. Если одновременно обнаружены несопоставленные central- и legacy-записи, адаптер считает это возможной ошибкой переименования и безопасно возвращается к локальному списку до добавления alias.

## Дополнительные зависимости

Установить только если модуль включается:

```powershell
python -m pip install -r VA/schedule_manager/requirements-va.txt
```

Отсутствие этих зависимостей не должно ломать основной Flask, если модуль выключен или папка удалена.

## Миграция данных

Миграция выполняется вручную:

```powershell
python -m VA.schedule_manager.tools.migrate_runtime_data --source "C:\schedule-manager-migration-20260714_163642" --dry-run
python -m VA.schedule_manager.tools.migrate_runtime_data --source "C:\schedule-manager-migration-20260714_163642"
```

Без `--overwrite` существующие runtime-файлы не заменяются.

## Удаление

Чтобы удалить модуль:

1. Выключить `modules.va_schedule_manager.enabled`.
2. Перезапустить Flask.
3. Удалить папку `VA/schedule_manager`.

Основной проект не должен импортировать VA-модуль на уровне корневых файлов, кроме optional registrar.
