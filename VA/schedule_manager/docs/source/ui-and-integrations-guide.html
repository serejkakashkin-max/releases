Как устроено приложение

Код организован вокруг одного глобального Flask-приложения, набора Blueprint и сервисных модулей. Для новой функции сохраняйте границу route → service → integration client, даже если соседний legacy-код смешивает эти роли.

Факт

Регистрация

app.py импортирует готовый app и передаёт его в единую функцию register_routes.

app.py:13–20; routes/__init__.py:4–37

Legacy / риск

Side effects

Startup выполняет filesystem I/O и запускает process-local worker. Импорт приложения поэтому нельзя считать чистой операцией.

extensions.py:14; routes/__init__.py:21–28

Рекомендация

Новый код не должен запускать поток, сеть или тяжёлое чтение при импорте. Если фоновой работе нужен lifecycle, сделайте явный idempotent starter, наблюдаемый status endpoint и внешний механизм остановки; для multi-process deployment используйте общую очередь/хранилище.

Слои конфигурации

Факт app.py:3–6 загружает .env с override; config.py:104–110 читает обязательную runtime-конфигурацию при импорте; services/feature_flags_service.py:368–419 обновляет file-backed flags по mtime; BASE_PATH несколько route-модулей читают отдельно.

Рекомендация Для новой интеграции создайте один typed settings loader/client factory. Не читайте секрет в template/JS и не размножайте historical config-key names.

Правило размещения новой функции

  1. Если функция однозначно принадлежит существующей области и модуль остаётся обозримым — добавьте route туда.
  2. Если функция имеет собственную страницу, API и сервисную границу — создайте новый Blueprint с явным prefix.
  3. Импортируйте и зарегистрируйте его только в register_routes; не вызывайте app.register_blueprint из service.
  4. Route отвечает за HTTP: request parsing, status, response. Service — за validation/business flow. Client — за внешний protocol.

import logging
import os
from uuid import uuid4

from flask import Blueprint, jsonify, render_template, request

from services.new_feature_service import InputError, ServiceTimeoutError, process_item


new_feature_bp = Blueprint("new_feature", __name__, url_prefix="/new-feature")
BASE_PATH = os.getenv("BASE_PATH", "").rstrip("/")
logger = logging.getLogger(__name__)


def error_response(code: str, message: str, status: int, request_id: str):
    return jsonify({
        "ok": False,
        "data": None,
        "error": {"code": code, "message": message},
        "request_id": request_id,
    }), status


@new_feature_bp.get("")
def new_feature_page():
    return render_template("new_feature.html", basepath=BASE_PATH)


@new_feature_bp.post("/api/run")
def run_new_feature():
    request_id = uuid4().hex
    payload = request.get_json(silent=True)
    if not isinstance(payload, dict):
        return error_response("INVALID_JSON", "Ожидался JSON-объект.", 400, request_id)

    try:
        result = process_item(payload.get("query"))
        logger.info("new_feature completed request_id=%s item_count=%d", request_id, len(result["items"]))
        return jsonify({"ok": True, "data": result, "error": None, "request_id": request_id})
    except InputError as exc:
        return error_response("VALIDATION_ERROR", str(exc), 400, request_id)
    except ServiceTimeoutError:
        logger.warning("new_feature timeout request_id=%s", request_id)
        return error_response("UPSTREAM_TIMEOUT", "Сервис не ответил вовремя.", 504, request_id)
    except Exception as exc:
        logger.error(
            "new_feature failed request_id=%s error_type=%s",
            request_id,
            type(exc).__name__,
        )
        return error_response("INTERNAL_ERROR", "Не удалось выполнить операцию.", 500, request_id)

В log попадают request ID, тип ошибки и количество элементов, но не traceback, query, token, payload или upstream body. Integration client обязан преобразовать и редактировать свои ошибки до route. Пользователю не возвращается str(exc) для неожиданных исключений.

(function () {
  'use strict';

  var config = window.NEW_FEATURE_CONFIG || {};
  var basePath = String(config.basePath || '').replace(/\/$/, '');
  var form = document.getElementById('newFeatureForm');
  var input = document.getElementById('query');
  var submit = document.getElementById('submitButton');
  var statusPanel = document.getElementById('statusPanel');
  var statusText = document.getElementById('statusText');
  var retry = document.getElementById('retryButton');
  var results = document.getElementById('results');
  var themeToggle = document.getElementById('themeToggle');
  var lastPayload = null;
  var inFlight = false;

  function showStatus(message, canRetry) {
    statusPanel.hidden = false;
    statusText.textContent = message;
    retry.hidden = !canRetry;
  }

  function renderItems(items) {
    results.replaceChildren();
    results.hidden = false;
    if (!items.length) {
      results.textContent = 'Ничего не найдено.';
      return;
    }
    items.forEach(function (item) {
      var card = document.createElement('article');
      card.className = 'new-feature__result';
      card.textContent = item.title + ' — ' + item.status;
      results.appendChild(card);
    });
  }

  async function parseResponse(response) {
    var text = await response.text();
    var body = null;
    try { body = text ? JSON.parse(text) : null; } catch (_error) { body = null; }
    if (!response.ok || !body || body.ok !== true) {
      var message = body && body.error && body.error.message
        ? body.error.message : 'Сервер вернул неожиданный ответ.';
      var error = new Error(message);
      error.status = response.status;
      throw error;
    }
    return body;
  }

  async function run(payload) {
    if (inFlight) return;
    inFlight = true;
    lastPayload = payload;
    submit.disabled = true;
    statusPanel.setAttribute('aria-busy', 'true');
    results.hidden = true;
    showStatus('Выполняется…', false);

    var controller = new AbortController();
    var timer = window.setTimeout(function () { controller.abort(); }, 10000);
    try {
      var response = await fetch(basePath + '/new-feature/api/run', {
        method: 'POST',
        headers: {'Content-Type': 'application/json'},
        body: JSON.stringify(payload),
        signal: controller.signal
      });
      var body = await parseResponse(response);
      renderItems(body.data.items || []);
      showStatus('Готово.', false);
    } catch (error) {
      if (error.name === 'AbortError') showStatus('Время ожидания истекло.', true);
      else if (error.status === 409) showStatus('Данные изменились. Обновите состояние.', true);
      else showStatus(error.message || 'Не удалось выполнить запрос.', true);
    } finally {
      window.clearTimeout(timer);
      statusPanel.setAttribute('aria-busy', 'false');
      submit.disabled = false;
      inFlight = false;
    }
  }

  form.addEventListener('submit', function (event) {
    event.preventDefault();
    var query = input.value.trim();
    input.classList.toggle('is-invalid', !query);
    if (!query) { input.focus(); return; }
    run({query: query});
  });
  retry.addEventListener('click', function () { if (lastPayload) run(lastPayload); });
  themeToggle.addEventListener('click', function () {
    var next = document.documentElement.getAttribute('data-theme') === 'dark' ? 'light' : 'dark';
    document.documentElement.setAttribute('data-theme', next);
    try { localStorage.setItem('theme', next); } catch (_error) { /* file/private mode */ }
  });
}());

Практический слой

Как переиспользовать существующие интеграции в новой функции

Базовая цепочка нового кода: route → feature service → существующий integration service/client. Route валидирует HTTP-вход и формирует стабильный ответ; feature service реализует сценарий; существующая граница владеет сетью, файлами и внешним контрактом.

Не дублировать

Запрещено создавать второй HTTP/mail-клиент, копировать токены, JQL, TLS-настройки, retry/error mapping или state-файлы. Route не выполняет raw requests, не читает token, не строит JQL, не работает с SMTP/IMAP, сертификатами и state-файлами и не импортирует private-функции с префиксом _.

Как добавить новую интеграцию

  1. Назначение и owner: сформулируйте operation, data classification, SLA и ответственного.
  2. Configuration: введите нейтральные env names; default допустим только для несекретных значений.
  3. Secret storage: token/password берётся из deployment secret store, не из Git, HTML или client JS.
  4. TLS/CA: verification включён; custom CA задаётся CA_BUNDLE. verify=False запрещён.
  5. Timeout: раздельные connect/read limits; бесконечное ожидание запрещено.
  6. Retries: bounded exponential backoff только для transient и idempotent calls; учитывать Retry-After.
  7. Idempotency: mutation получает idempotency key или deduplication contract.
  8. Validation: allowlist, size/type/schema до network; upstream response валидируется отдельно.
  9. Error contract: client mapping → domain exception → route code/status → понятный UI message.
  10. Logging/redaction: request ID, operation, duration/status; без token, auth header, payload и персональных полей.
  11. Route/service boundary: route разбирает HTTP и формирует response; service управляет use case; client знает transport и не возвращает Flask response.
  12. UI states: loading, empty, timeout, offline, retry, partial/degraded и safe disable path.
  13. Testing: unit validation/mapping, mocked client, timeout/TLS/429/5xx, route contract и UI smoke.
  14. Feature flag: disabled-by-default для рискованного rollout; flag не заменяет access control.
  15. Rollback: заранее описать queue drain/compensation, config rollback и data compatibility.
  16. Отключение: предусмотреть независимый kill switch, понятный degraded UI и проверку, что disabled client не делает network calls.

Git как система, review как процесс

Рекомендуемый базовый workflow

Git-команды одинаковы для GitHub и Bitbucket. Pull Request (GitHub) и Pull/Merge Request в других hosting UI — механизм review, а не часть самой системы Git.

  1. Проверьте рабочую копию, затем обновите main: git status --short --branch, git fetch --prune origin, git switch main, git pull --ff-only origin main.
  2. Создайте рабочую ветку от актуальной main: git switch -c <feature-branch>. Формат имени определяет команда.
  3. Делайте небольшие осмысленные commits; не добавляйте secrets/generated personal data.
  4. Перед review обновите feature-ветку одним из вариантов ниже и прогоните проверки.
  5. Откройте PR/MR, опишите scope, endpoints, UI states, config, tests, rollout и rollback.
  6. После approvals и успешных checks выполните merge выбранной командой стратегией.
  7. Обновите локальную main, сравните SHA с remote, повторите smoke/checks и проверьте фактический результат после доставки.
Базовый вариант

Merge актуальной main в feature

git fetch origin
git switch <feature-branch>
git merge origin/main
# решить конфликты, проверить, push обычным способом

История не переписывается; безопаснее и понятнее для совместной опубликованной ветки.

Продвинутый

Rebase на актуальную main

git fetch origin
git switch <feature-branch>
git rebase origin/main
# для уже опубликованной ветки, только после координации:
git push --force-with-lease

Переписывает commit IDs. Использовать сотрудникам, понимающим последствия, и только по командной договорённости.

Запрет

Обычный git push --force не использовать. --force-with-lease допустим только для осознанного rebase опубликованной feature-ветки после координации; защищённые/shared ветки не переписывать.
